## What this is

Orkai is a self-hosted PaaS that runs on **K3s** (single-binary Kubernetes). Unlike Coolify/Dokploy, it does not wrap Docker ‚Äî every workload (apps, databases, cron jobs) is a real Kubernetes object, so `kubectl`/`helm` work directly against it. The product surface: git-push/Docker-image app deploys with in-cluster Kaniko builds, managed databases, K8s CronJobs, cluster inspection (nodes, Helm, DaemonSets, Traefik), RBAC, 2FA, and multi-channel notifications.

> The README and Makefile call the web app "Next.js" ‚Äî this is stale; it is **Vite + React 19 + TanStack Router**.

## Monorepo layout

Bun workspaces (`apps/*`, `packages/*`). Two apps:
- `apps/api` ‚Äî Go 1.25 backend (Gin, Bun ORM, K8s client-go)
- `apps/web` ‚Äî Vite + React 19 SPA

## Commands

All via the root `Makefile`:

```bash
make dev            # docker-up + run API (air hot-reload) and web (bun) together
make dev-api        # Go API only, hot reload via `air`
make dev-web        # web only (vite, port 3000)
make build          # build api binaries (server/worker/migrate) + web
make fmt            # gofmt+goimports on Go, biome --write on web
make lint           # golangci-lint + biome check
make test           # go test ./... + bun test
make test-api       # go test -race -cover ./...
make docker-up      # local infra: Postgres, Valkey, K3s (deploy/docker-compose.yml)
```

Single Go test: `cd apps/api && go test ./internal/service -run TestName`.
Migrations: `make migrate` / `make migrate-rollback` / `make migrate-status` / `make migrate-create name=add_users`.
Web build alone: `cd apps/web && bun run build`.
OpenAPI docs: `go install github.com/swaggo/swag/cmd/swag@latest` then `make gen-api` (generates spec for apps/docs; no runtime swagger on the API).

Formatting is **Biome** (2-space, 100 col, configured in `biome.json`). `make fmt` must pass before pushing.

## Backend architecture (`apps/api`)

Three entrypoints under `cmd/`: `server` (the API, this is what runs), `worker` (PGMQ consumer ‚Äî **stub/TODO**, not yet implemented), `migrate` (CLI for migrations). The `server` binary auto-runs migrations on boot (guarded by a Postgres advisory lock).

Layered, dependency-injected. Wiring happens in `cmd/server/main.go` ‚Üí `service.NewContainer(...)` ‚Üí `server.NewRouter(...)`. The layers:

- **`internal/api/v1`** ‚Äî Gin HTTP handlers (one file per domain: `app_handler.go`, etc.). Thin; parse/validate ‚Üí call a service.
- **`internal/api/ws`** ‚Äî WebSocket + SSE handlers: container logs, node logs, web terminal (exec), and the `/ws/events` SSE broker. WS auth is via `?token=` query param (`middleware.WSAuth`).
- **`internal/api/middleware`** ‚Äî auth (JWT), CORS, rate limiting, request ID, recovery, Sentry, branding.
- **`internal/service`** ‚Äî business logic. All services are constructed into a `Container` (`service.go`). Services depend on `store.Store`, `orchestrator.Orchestrator`, and each other (e.g. `Deploy` depends on `Build` + `Notification`).
- **`internal/orchestrator`** ‚Äî the key abstraction. `Orchestrator` is a large composite interface (`AppManager`, `DatabaseManager`, `IngressManager`, `BuildManager`, etc.). `k3s/` is the real implementation (one file per concern: `deploy.go`, `build.go`, `ingress.go`, `database.go`, `helm.go`, `exec.go`, ‚Ä¶). `noop.go` is a fallback used when no kubeconfig is present, so the API runs without a cluster. **Anything that touches Kubernetes goes through this interface ‚Äî never call client-go from services directly.**
- **`internal/store`** ‚Äî persistence. `store.Store` interface; `store/pg/` is the Bun/Postgres implementation (one store file per entity). `metrics_store.go` is separate (same DB, independent).
- **`internal/model`** ‚Äî Bun ORM structs.
- **`internal/event`** ‚Äî real-time pipeline: Postgres `LISTEN/NOTIFY` ‚Üí `PGSubscriber` ‚Üí `SSEBroker` ‚Üí browser. This is how the UI gets live updates.

### Request ‚Üí cluster flow
HTTP handler ‚Üí service ‚Üí orchestrator (k3s) ‚Üí Kubernetes. State persisted via store. Live status/logs flow back through the WS/SSE layer.

### Migrations
Bun migrations in `apps/api/migrations/`, named `YYYYMMDD_NNN_description.go`. Each file **self-registers via `init()`** onto the shared `Migrations` collection in `migrations.go`. `make migrate-create name=...` scaffolds a new one.

## Frontend architecture (`apps/web/src`)

- **Routing**: TanStack Router with **file-based routes** in `routes/`. `routeTree.gen.ts` is generated by the vite plugin ‚Äî do not edit by hand. `_dashboard.tsx` is the authed layout; `_dashboard/` holds top-level pages; `projects_/$id/` holds the per-project detail view, whose tabs live in `_components/-*.tsx` (the `-` prefix excludes them from routing).
- **Data**: TanStack Query. One hook per domain in `hooks/` (`use-apps.ts`, `use-cluster.ts`, ‚Ä¶) wrapping the shared `ApiClient` in `lib/api.ts`. The client injects the JWT bearer and throws `UnauthorizedError` on 401 (caught to redirect to login).
- **Live updates**: `hooks/use-event-source.ts` consumes the `/ws/events` SSE stream and invalidates queries.
- **Dev proxy**: vite proxies `/api` and `/ws` ‚Üí `localhost:8080` (the Go API). Path alias `@` ‚Üí `src/`.
- UI: Tailwind v4 + Radix primitives (`components/ui/`), Recharts, `@xyflow/react` for node topology.

## Conventions

- Branch from `main`, one focused change per commit, keep PRs small (see `CONTRIBUTING.md`).
- New backend feature typically touches all layers: model ‚Üí store ‚Üí service ‚Üí handler ‚Üí router registration, plus the orchestrator interface + k3s impl if it hits the cluster.
- New frontend feature: a `hooks/use-*.ts` query hook + a route/component; never edit `routeTree.gen.ts`.
